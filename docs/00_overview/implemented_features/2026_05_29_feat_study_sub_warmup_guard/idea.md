# Study budget sub-warmup guard ("don't ship a study the optimizer never woke up for")

**Date:** 2026-05-29 (preflight-refreshed 2026-05-29 after [`chore_study_default_stop_conditions`](../../../implemented_features/2026_05_23_chore_study_default_stop_conditions/) was confirmed shipped 2026-05-23)
**Status:** Idea — surfaced from an operator dogfooding review of MVP1 studies (2026-05-29). The preset-defaults half of the original diagnosis already shipped via [`chore_study_default_stop_conditions`](../../../implemented_features/2026_05_23_chore_study_default_stop_conditions/) (Focused 50 / Standard 200 / Deep 1000 / Custom; default `max_trials=200` pre-fill; refreshed glossary + system prompt). Remaining scope: a **Custom-mode sub-warmup inline warning** + an **optional digest narrative note** when a completed study fell below the warmup floor.
**Priority:** P1 (for the "overnight autopilot ergonomics" theme) — the *guard* is the high-leverage half of the original idea. Without it, the Custom escape hatch lets operators silently re-introduce the exact under-budgeting failure mode the shipped presets prevent for the default path.
**Origin:** Operator ran 7 studies through the full loop and felt follow-ups were near-mandatory. Tracing the actual `studies.config` values (2026-05-29) showed the cause: 6 of 7 studies ran `max_trials` of **12–15** (these studies all pre-dated the 2026-05-23 default-presets PR, so the wizard offered no pre-fill at the time). With Optuna TPE's ~10-trial random warmup ([`optimization.md`](../../../../01_architecture/optimization.md) §"Optuna configuration"; `n_startup_trials` defaulted in [`backend/app/eval/optuna_runtime.py:107`](../../../../../backend/app/eval/optuna_runtime.py#L107)), those studies did ~10 random samples + 2 TPE-guided trials + 1 baseline — the Bayesian optimizer barely engaged before the stop condition fired. The unconverged result is what made the digest emit "narrow/widen" follow-ups ([`backend/app/llm/digest_prompt.py:121`](../../../../../backend/app/llm/digest_prompt.py#L121)), which *felt* like a required second pass but was really compensation for under-budgeting. The shipped presets close the no-pre-fill path; this idea closes the Custom-mode path.
**Depends on:** MVP1 study lifecycle (shipped) + [`chore_study_default_stop_conditions`](../../../implemented_features/2026_05_23_chore_study_default_stop_conditions/) (shipped 2026-05-23 — establishes the shared 50/200/1000 preset surface and the MedianPruner-activation floor at 50 trials that this guard piggybacks on). No dependency on the MVP2 anchors; composes with them.

## Problem

The original idea (pre-preflight) named three connected gaps. After the 2026-05-23 chore landed, **two are closed for the default path**; one remains open everywhere and a second remains open in the Custom escape hatch:

1. **~~No trial-budget default~~ — Closed for default path; open in Custom mode.** `StudyConfigSpec` still requires the operator to supply `max_trials` or `time_budget_min` ([`backend/app/api/v1/schemas.py:629`](../../../../../backend/app/api/v1/schemas.py#L629)) — that server-side guard was intentionally kept. The wizard now pre-fills `max_trials=200` and offers Focused (50) / Standard (200) / Deep (1000) / Custom presets ([`ui/src/components/studies/create-study-modal.tsx:91-110`](../../../../../ui/src/components/studies/create-study-modal.tsx#L91-L110)). The Custom path still lets an operator type any number ≥ 1, including 12–15.

2. **No floor at the TPE warmup threshold (still open for Custom).** Optuna's `TPESampler` runs its first ~10 trials as random search before the estimator guides sampling (Optuna default `n_startup_trials=10`; the runtime at [`backend/app/eval/optuna_runtime.py:107`](../../../../../backend/app/eval/optuna_runtime.py#L107) does not override it); `MedianPruner` is disabled under 50 trials ([`backend/app/eval/optuna_runtime.py:116-156`](../../../../../backend/app/eval/optuna_runtime.py#L116-L156), [`optimization.md`](../../../../01_architecture/optimization.md)). A study with `max_trials < ~20` therefore barely exercises the Bayesian optimizer at all — it is effectively random search. The shipped presets all clear this floor; only Custom can re-introduce the regression, and nothing in the form warns the operator at edit time.

3. **The cost of under-budgeting is invisible and misattributed (still open).** The unconverged study produces a digest with "narrow"/"widen" follow-ups ([`backend/app/llm/digest_prompt.py:121`](../../../../../backend/app/llm/digest_prompt.py#L121)). The operator reasonably reads that as "the tool needs me to iterate," when the real signal is "this study stopped before the optimizer learned anything." This misattribution lands the same way whether the under-budget happened in Custom mode today or in a pre-2026-05-23 study; closing it is the digest-narrative half of this idea (and may belong to [`feat_study_convergence_indicator`](../feat_study_convergence_indicator/idea.md) instead — see open question §3).

**Evidence (traced 2026-05-29 against the live DB; all 7 studies pre-date the 2026-05-23 preset PR):** 7 studies total; `max_trials` distribution = `12 (×4), 15 (×2), 200 (×1)`; zero used `auto_followup_depth`; zero chain children. The single 200-trial study is the only one that would have converged. Today, a fresh operator running the wizard inherits `max_trials=200` (Standard preset) by default; the 12–15 distribution would only recur if they explicitly switch to Custom and pick a low number — which is exactly what this guard intercepts.

## Proposed capabilities

### ~~Budget presets in the create-study wizard~~ — shipped 2026-05-23

Already delivered by [`chore_study_default_stop_conditions`](../../../implemented_features/2026_05_23_chore_study_default_stop_conditions/feature_spec.md): Focused (50) / Standard (200) / Deep (1000 + 480-min cap) / Custom, with `max_trials=200` as the form default, an `InfoTooltip` on `study.preset` ([`ui/src/lib/glossary.ts:172-176`](../../../../../ui/src/lib/glossary.ts#L172-L176)), and the system prompt at [`prompts/orchestrator.system.md`](../../../../../prompts/orchestrator.system.md) updated to teach the agent the same numbers. The shipped presets are calibrated to the MedianPruner activation threshold (Focused=50 sits exactly at the threshold per [`optuna_runtime.py:116-156`](../../../../../backend/app/eval/optuna_runtime.py#L116-L156)). **No further preset work is in this idea's scope.**

### Sub-warmup guard for Custom mode (the net-new core)

When an operator switches the Stop-condition group to **Custom** and enters a `max_trials` below the warmup-derived floor (recommended `< 50`, which matches the MedianPruner activation threshold the rest of the system already keys off — see open question §1 for the trigger-value decision), surface a non-blocking inline warning under the `max_trials` input at create time:

> "TPE runs its first ~10 trials as random search before it starts optimizing. At 12 trials this study is essentially random search and is unlikely to converge — switch to Focused (50) or Standard (200) for a result worth turning into a PR."

Non-blocking (the operator can still proceed — quick smoke tests are legitimate), but it makes the cost legible at the moment of the decision. This is the single behavior change that would most have helped the operators whose 12-trial studies the evidence traced. Belongs **only** in the Custom-mode path — the three non-Custom presets all clear the floor by construction and don't need the warning.

### Optional: surface the warmup boundary in the digest

When a completed study ran fewer trials than the warmup floor (`trials_run < 50`), the digest's narrative (or a small banner on the proposal) notes that the result is pre-convergence and that the right next step is *re-running with a larger budget*, not necessarily accepting a narrow/widen follow-up. This stops the misattribution at the point where the operator reads the result.

This capability **may instead belong to [`feat_study_convergence_indicator`](../feat_study_convergence_indicator/idea.md)** — see open question §3 for the ownership call. If `feat_study_convergence_indicator` claims it, drop it from this idea's scope and ship only the Custom-mode guard here.

## Why not just tell people to set max_trials higher?

Because the data shows they won't, and the tool gives them no reason to. The whole point of the Karpathy-loop framing is that the operator shouldn't have to be a Bayesian-optimization expert to get a converged result — "tireless and structured" is the *tool's* job ([blog: haystack-to-relyloop](../../../../blog/2026-05-20-haystack-to-relyloop.md)). A defaulted, warmup-aware budget is the cheapest way to make the loop deliver on that promise.

## Open questions for /spec-gen

1. **Custom-mode warning trigger threshold.** Recommended default: **`< 50`** — matches the MedianPruner activation floor at [`optuna_runtime.py:121`](../../../../../backend/app/eval/optuna_runtime.py#L121), reuses a constant already canonical in shipped code, and aligns with the Focused preset (50) so the warning's recommended remediation ("switch to Focused") lines up with the threshold. Alternative considered: `< 20` (= `2 × n_startup_trials`) from the idea's original sketch — tighter but introduces a second magic number no other surface uses. **Lock unless `/spec-gen` finds an operator-research reason to differ.**
2. **Warning text + remediation phrasing.** Stays a `/spec-gen` decision because it's user-facing copy, but two anchors: (a) name the specific Focused/Standard presets (so the operator has a one-click path back to a safe budget), (b) do not block submit (smoke tests against template wiring are a legitimate use of low-trial Custom runs).
3. **Digest convergence-note ownership.** Genuine open question — needs a product call. Recommendation: **own it in [`feat_study_convergence_indicator`](../feat_study_convergence_indicator/idea.md)** rather than here, because (i) the convergence-indicator feature already owns the "did this study plateau?" framing end-to-end, (ii) this idea's Custom-mode guard is purely create-time UX, and (iii) splitting create-time + result-time guards across two specs lets each be sized and tested independently. If `feat_study_convergence_indicator` declines, fall back to owning the digest note here as a small additive branch in `digest_prompt.py`. **Decide before `/spec-gen` writes either feature.**
4. **Should the same guard fire for the agent-driven `create_study` tool?** The system prompt at [`prompts/orchestrator.system.md`](../../../../../prompts/orchestrator.system.md) already teaches `max_trials=200` (Standard) as the recommended default, so the agent rarely picks a sub-warmup number on its own — but it *can*, and there is no server-side guard. Recommendation: out of scope for this idea (keep it to the wizard surface where the operator actually sees the warning); revisit if telemetry shows the agent emitting sub-50 trial counts.
