**Priority:** P2

**Depends on:** None. Self-contained migration. **Sequencing-pressure update (preflight 2026-05-22):** [`feat_pr_metric_confidence`](../../../00_overview/implemented_features/2026_05_21_feat_pr_metric_confidence/) Phase 1 has already shipped, and inspection of [`backend/app/domain/study/confidence.py:247`](../../../../backend/app/domain/study/confidence.py) confirms `bootstrap_ci_95()` consumes the `per_query_metrics` dict (user-facing keys, not pytrec_eval wire forms) — it does NOT import pytrec_eval directly. The "land before further confidence surface grows" pressure still applies for the **next** wave (paired-comparison + Fisher randomization), which has not shipped yet; this migration unblocks those features sitting cleanly on `ir_measures.iter_calc()` per-query output rather than the existing scoring re-keying.

## Problem

`pytrec_eval` is unmaintained — the canonical repo's last commit is 2020-09-07 and there have been zero GitHub releases. The package still works (it's a thin C wrapper over the stable `trec_eval` binary), but we're carrying long-term risk: no Python 3.14+ wheels are guaranteed, no fixes for upstream `trec_eval` bug reports, no maintainer to file issues against, and a C-extension build that has historically caused Apple-Silicon / Alpine / arm64 install pain. `ir_measures` (from the PyTerrier team) wraps `pytrec_eval` + `gdeval` + `judged-as-relevant` behind a **provider interface**, gives us a typed metric DSL (`nDCG@10`, `AP@5`, etc.), exposes per-query iteration via `iter_calc()` for downstream bootstrapping, and has active maintenance with Python 3.13 wheels. Replacing the direct `pytrec_eval` import with `ir_measures` insulates the scoring layer from the abandoned-upstream risk without forcing another full rewrite the next time a backend becomes preferable — swapping providers becomes a config change rather than an `import` change.

## Proposed capabilities

### Capability 1 — Swap `pytrec_eval` for `ir_measures` in the scoring layer

- Replace `import pytrec_eval` with `import ir_measures` plus the metric-object DSL imports (`from ir_measures import nDCG, AP, RR, P, R`) in [`backend/app/eval/scoring.py`](../../../../backend/app/eval/scoring.py).
- Keep `_translate_metric_name()` but simplify it — it now maps `("ndcg", 10)` → `nDCG@10` (the `ir_measures` metric object) instead of the `"ndcg_cut_10"` wire string. The translation table is smaller (one entry per metric family, with `k` applied via the `metric @ k` operator) but does not disappear entirely. `objective_metric_key()` survives unchanged.
- Preserve the public `score(qrels, run, metrics) -> ScoreResult` signature exactly so callers (the trial worker + tests) need no changes. The re-keying step that converts the backend's output back to the user-facing form (`ndcg@10`, `map@5`) stays — `trials.metrics` JSONB depends on those keys.
- Update the `SUPPORTED_METRICS` / `SUPPORTED_K_VALUES` docstring to cite `ir_measures`' metric vocabulary as source of truth.
- Update [`pyproject.toml`](../../../../pyproject.toml) — drop `pytrec-eval>=0.5` from the direct pins, add `ir-measures>=0.4.3`, drop the `[[tool.mypy.overrides]]` block for `pytrec_eval` (`ir_measures` ships type hints; `pytrec_eval` is no longer directly imported, so even if it remains a transitive backend its missing stubs no longer leak into our type-check surface).

> **Open question for impl-plan time:** does `ir-measures>=0.4.3` keep `pytrec_eval` as a default transitive backend for the metrics in `SUPPORTED_METRICS × SUPPORTED_K_VALUES`, or do its other built-in providers cover them all? If the C extension still ships transitively, the [`Dockerfile`](../../../../Dockerfile) stage-2 `gcc` / `g++` / `python3-dev` install (lines 44–54) stays as-is; if it can be dropped, that's a nice ergonomic win on arm64 / Apple Silicon first-builds. The verification is a one-shot `pip install ir-measures && pip show pytrec_eval` (or equivalent in `uv tree`). Resolve before drafting Capability 1 in the implementation plan; do **not** drop the Dockerfile block speculatively.

## Why deferred

Not blocking anything today: `pytrec_eval` still installs and produces correct numbers. **The "land before feat_pr_metric_confidence ships" framing is obsolete** — that feature's Phase 1 already shipped 2026-05-21 (PR #180), and the shipped bootstrap CI at [`backend/app/domain/study/confidence.py:247`](../../../../backend/app/domain/study/confidence.py) consumes user-facing per-query metric keys, not pytrec_eval wire forms — so this migration won't have to rewrite Phase 1 code. The remaining sequencing pressure is the **next** confidence wave (paired-comparison + Fisher randomization, neither shipped yet); queueing this migration ahead of that wave keeps the new statistical code born on `ir_measures.iter_calc()` from day one. P2 cleanup otherwise.

Rejected on (a) **single-maintainer bus factor** — same failure mode that triggered this migration in the first place; swapping one abandoned-upstream risk for another doesn't improve anything, (b) **Numba install + cold-start cost** — ~30 MB wheel plus first-import JIT warm-up we'd be paying on every API container boot, (c) **no provider abstraction** — if `ranx` is ever abandoned we'd need another full rewrite, whereas with `ir_measures` swapping the backend is a config change. `ranx` remains available as an optional `ir-measures[ranx]` backend if its bootstrap / fusion features are needed later.
