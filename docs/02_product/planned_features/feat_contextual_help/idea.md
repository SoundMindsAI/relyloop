## Problem

RelyLoop's UI is dense with IR (information-retrieval) and Optuna concepts — NDCG@10, MAP, MRR, ERR, K, TPE sampler, median pruner, judgment scale 0–3, Cohen's κ, study state machine, proposal lifecycle, PR state, cluster auth kinds, Docker-secrets credential refs, query templates, search-space JSON — and none of them are explained where they appear. A search relevance engineer encountering the tool for the first time faces a steep onboarding cliff: the [tutorial](../../../08_guides/tutorial-first-study.md) walks them through Study #1, but the moment they tab over to the create-study modal to set up Study #2 they hit a five-step form with six unlabeled metric/sampler/pruner/seed/parallelism knobs and no per-field guidance. The result is that the [tutorial](../../../08_guides/tutorial-first-study.md) is load-bearing in a way it shouldn't be — the UI itself should carry most of the explanation.

The audit identified **30+ specific opportunities** across eight surfaces. Highest-leverage cluster: the create-study modal (six obscure fields in one step) and the digest panel on the study detail page (a parameter-importance chart, a metric delta percentage, and LLM-suggested follow-ups, all with no legend or interpretation guidance).

## Proposed capabilities

Three reusable IA patterns, applied phased across the surfaces ranked by user impact:

### Capability 1 — `<InfoTooltip>` primitive + wrapper

- Add `@radix-ui/react-tooltip` to `ui/package.json` and ship `ui/src/components/ui/tooltip.tsx` following the same shadcn shape as the existing primitives. This is the first tooltip primitive in the codebase (verified empty `grep` for tooltip in `ui/src/components/ui/`).
- Ship a thin `<InfoTooltip label="...">{children}</InfoTooltip>` wrapper in `ui/src/components/common/info-tooltip.tsx` that renders an `<Info />` icon trigger (from `lucide-react`, the icon set already used by [`ui/components/ui/dialog.tsx`](../../../../ui/src/components/ui/dialog.tsx) and [`ui/components/ui/select.tsx`](../../../../ui/src/components/ui/select.tsx)) plus tooltip body. Single canonical visual treatment so every label-adjacent help icon looks identical across the UI.
- Used for **short** definitions: status meanings, metric direction (higher = better vs. lower = better), rating-scale legends, units reminders. Anything ≤ ~140 chars of body text.

### Capability 2 — `<HelpPopover>` for dense concepts

- Existing `Popover` is already used at [override-popover.tsx](../../../../ui/src/components/judgments/override-popover.tsx) and [edit-query-popover.tsx](../../../../ui/src/components/query-sets/edit-query-popover.tsx), but only for input forms — never as help affordance. Ship `ui/src/components/common/help-popover.tsx` styled as a help target: same Info icon trigger as `InfoTooltip`, but with a wider body and support for markdown/list content.
- Used for **longer** explanations that don't fit in a tooltip: "what's a TPE sampler vs. random?", "when should I fork a template?", "what does a κ value of 0.65 mean?", "metric definitions for NDCG/MAP/MRR/ERR with one-line use-case guidance each".
- The visual contract is: clicking an info icon shows a tooltip (short fact) OR a popover (longer body). Never both on the same icon — the wrapper a label uses dictates which.

### Capability 3 — onboarding nudges + empty-state copy *(Phase 3 only — out of MVP1 spec scope)*

- Three surfaces need true onboarding-grade copy, not just per-field tooltips:
  1. **Home page** (`ui/src/app/page.tsx`) — when no clusters / no studies / no proposals exist, show a numbered "start here" panel with three steps linking to the right pages. Today the count cards render `0` with no guidance.
  2. **`/clusters` empty state** — explain the Docker-secrets credential mounting mechanism inline; today the only hint is the "Credentials ref" label, which reads `./secrets/<name>` with no context for a first-time operator.
  3. **`/chat` first-message state** — render 3–5 example prompts ("Tell me about the prod-es cluster", "Run a study optimizing NDCG@10 for the product-search index", "Open a PR for the latest proposal") as click-to-send chips so a new user has something concrete to try.

## Why now (MVP1 Phase 1)

Alpha is open and the [release checklist](../../../03_runbooks/release-checklist.md) §8 feedback Discussion + design-partner channel shares are imminent. The tutorial is currently the only onboarding surface, and the create-study modal is the steepest cliff a design partner hits the moment they step off the tutorial path — six unlabeled metric/sampler/pruner/seed/parallelism knobs in one form step, with no per-field guidance. Every day of partner feedback that reads "I didn't know what TPE meant" or "what does pruned mean?" is feedback Phase 1 tooltips would have absorbed.

Phase 1 is scoped tight: Capability 1 + 2 primitives (~150 LOC of new infra) plus the create-study modal + study-detail surface. Phases 2 (judgments + proposals) and 3 (chat + cluster registration + home onboarding) stay in this idea as documented future scope but are not part of the MVP1 spec — they'll be picked up under a separate MVP2 idea after design-partner feedback surfaces which surfaces matter most.

## Locked decisions

- **Scope: MVP1 Phase 1 only.** `/spec-gen` covers Phase 1 (create-study modal + study detail). Phases 2 (judgments + proposals) and 3 (chat + cluster registration + home onboarding) remain documented in this idea as future scope but are explicitly NOT in the spec generated from this file. A follow-up idea picks them up in MVP2 after design-partner feedback surfaces priorities. Folder stays `feat_contextual_help` (no `_mvp2` rename — Phase 1 ships in MVP1). The "start here" panel design decision (Stripe-style checklist vs. illustration vs. simple ordered list) is deferred to the future MVP2 idea, not locked here.
- **Glossary centralization.** Ship `ui/src/lib/glossary.ts` as the single source of truth for tooltip/popover copy from day one, even though Phase 1 only populates the create-study + study-detail subset. CLAUDE.md's "Enumerated Value Contract Discipline" already requires `// Values must match <backend file>` comments above frontend option arrays; the glossary file is the natural place to attach those same comments to copy that *describes* the enum values. Inlining copy per-component would re-introduce the drift risk the rule was written to prevent. Each glossary entry pairs a wire value (e.g., `"completed"`, `"ndcg"`, `"es_apikey"`) with: short tooltip copy, longer popover body (where applicable), and a path comment to the backend source-of-truth file.
- **`<Info />` icon source.** Use `lucide-react`'s `Info` icon, matching the existing precedent set by [`ui/components/ui/dialog.tsx:3`](../../../../ui/src/components/ui/dialog.tsx) (`X`) and [`ui/components/ui/select.tsx:3`](../../../../ui/src/components/ui/select.tsx) (`Check, ChevronDown`). No new icon dependency.
- **Tooltip vs. popover usage rule.** `InfoTooltip` for ≤ ~140 chars (fact, definition, unit, status meaning). `HelpPopover` for longer body (multi-line guidance, interpretation rubric, multi-option comparison). A single info icon never escalates from tooltip to popover at runtime — the wrapper the component author chooses dictates which.
